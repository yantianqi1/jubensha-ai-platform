# 剧本杀玩法与 LLM 融合研究

## 研究起点

按你给出的两份文档摘要，现有方案已经形成了一个很清晰但并不完整的分工：创作期把 LLM 当作剧本与世界观生产工具；单人运行期把 LLM 推到台前，承担 AI DM、NPC 扮演和生成式 UI；多人运行期则尽量回到权威状态机、房间管理和信息分发，让 LLM 退出裁判席。真正悬而未决的，不是“要不要用 LLM”，而是**LLM 的表达能力怎样和可验证的游戏状态机合并**。

这恰好也是当下学术界在 Murder Mystery Games 与交互叙事里最难的部分。PLAYER* 把 MMG 定义为一种同时要求自然语言推理、策略性问话、嫌疑缩减、且**状态空间未定义、缺少中间奖励**的复杂社会推理环境；ACL 2026 的另一篇工作则进一步把 MMG 看成**不完美信息、带欺骗性的多人推理任务**，并用协作式多 Agent 脚本生成、CoT 微调和 GRPO 来处理“角色身份差异、隐藏事实与误导信息”这些问题。换句话说，业界文档里留白的“运行时融合”，恰恰就是研究前沿最难啃的骨头。citeturn8view1turn8view0

因此，这份报告的核心判断很直接：**剧本杀里最值得让 LLM 做的，是“演、讲、诱导、氛围、解释、个性化”；最不该直接交给 LLM 的，是“解锁、判定、结算、权限、真值存储”**。这不是保守，而是因为多项研究都已经表明，LLM 在叙事和交互上可以很强，但在长期、可验证、可审计的规则一致性上仍然脆弱。RPGBench 明确指出，当前先进模型能讲出有意思的故事，却仍然会在**长程、复杂场景中的一致性与可验证机制**上失手；OpenAI 的 Structured Outputs 也明确提醒：即使输出严格符合 JSON Schema，模型仍可能在字段值里犯“语义错误”。citeturn25view0turn12search1turn12search2

## 网络上的剧本杀玩法图谱

今天网络上的“剧本杀”已经不是单一产品，而是一组分化很明显的玩法集合。中国主流线上平台并不只是把纸本搬上网，而是已经把**语音房、实时匹配、场景搜证、私聊、技能/机关、职业 DM 带本、围观**等能力捏在一起：百变大侦探强调“语音连麦、场景式搜证、私聊、剧情与技能机关玩法、专业 DM 与围观”；我是谜强调“海量剧本、即时匹配、稳定即时语音”；成森守姬则把“线上 DM 鲜活演绎、系统说书人、公证、全时段语音互动、随机匹配”作为卖点。这说明线上剧本杀的真实产品形态，本质上已经是**社交语音 + 线索系统 + 主持调度 + 轻规则服务端**的混合体。citeturn3search1turn3search3turn3search0turn3search2turn30search2

从内容类型看，中文语境里较稳定的分类包括推理本、还原本、情感本、欢乐本、阵营本、机制本、恐怖本、演绎本；而玩家需求大体对应脑力锻炼、情绪宣泄、社交合作/对抗、刺激体验四类。也就是说，“剧本杀”这个词下面，实际上覆盖的是一组截然不同的体验诉求：有些本子主打逻辑压迫感，有些主打哭点，有些主打社交和拉帮结派，有些主打机关、小游戏和现场表演。LLM 的介入深度不能按“剧本杀”三个字统一设计，必须按**本型**来切。citeturn0search1turn0search2turn0search4turn1search3turn17search0

国际市场同样在分化，而且已经出现几条对你这个研究问题非常有启发的路线。第一条是**无主持或自主持派对型玩法**：Foulplay 把“host can play too”作为卖点，Getting Away With Murder 明确强调“no game-master”，并支持浏览器、Tabletop Simulator 和 Print & Play。第二条是**AI 主持型派对工具**：MysteryPartyNow 直接承诺 AI 生成线索、运行故事、评最终指控，并且能在玩家迟到或掉队时重写故事、重新分配线索；Mystery Games、Krimora、Alibi 也都在卖“AI 主持、即时建局、角色与脚本自动生成、按幕推进”的体验。第三条是**自然语言审问型侦探游戏**：Cluebound 让玩家用自然语言问 AI 嫌疑人，Arsenic 明确强调“不是对话树，而是开放盘问”，嫌疑人会说谎、闪躲、前后矛盾、在压力下崩盘。它们不是传统剧本杀，但它们已经把用户教育到“问什么都能答”“每局不一样”“没有固定对白树”这个方向上了。citeturn19search1turn19search3turn29search3turn29search2turn29search10turn29search11turn29search0turn29search7turn29search9

结合这些现象，可以把网络玩法粗略看成下表：

| 玩法路线 | 现在网络上的典型形态 | 适合 LLM 的深度 |
| --- | --- | --- |
| 线上语音房剧本杀 | 实时匹配、语音、搜证、私聊、职业 DM、围观 | 中等，偏“主持辅助”和“解释层” |
| 线下桌面/实景搜证/沉浸剧场 | 围桌讨论、实景找线索、NPC 演绎、复盘 | 中等，偏“NPC 演绎”和“后处理” |
| 无主持派对 | 规则清晰、主持也能参与、浏览器/打印即玩 | 中高，适合做“自动主持” |
| AI 主持派对 | 动态分配角色/线索、按幕控制、适应掉线/迟到 | 很高，适合做“AI DM” |
| 自然语言审问侦探 | 开放盘问、AI 嫌疑人、强自由问答 | 很高，适合做“单人审问核心玩法” |

这个图谱的含义很关键：**不是所有剧本杀都适合“AI 全接管”**。机制/阵营/硬规则结算越重，LLM 越应该退到旁白和解释层；审问、搜证、角色扮演、情绪收束越重，LLM 越应该走到台前。citeturn3search1turn30search2turn29search3turn29search2turn29search9

## 用户需求与 LLM 机会

从用户视角看，剧本杀被购买的从来不只是“找凶手”。针对青年玩家的研究把剧本杀里的核心收益概括为**情绪价值、情绪释放、身体参与**；另一项心理学研究则强调，剧本杀最大的特色是“扮演角色去互动表演”，它给用户提供的是一种自我表达与心理满足。更落地的一项网络文本分析进一步发现，线下玩家最关注的高频核心，不是“答案”，而是“**剧本—主持人—角色**”这个三角结构；负面体验主要来自剧本质量低、环境差、主持人专业度不足。换句话说，玩家买的并不只是谜题，而是一个**角色被托住、节奏被托住、情绪被托住**的游玩过程。citeturn17search0turn17search1turn20search0

这组发现对 LLM 的启发非常直接。LLM 最擅长的并不是做“唯一正确判定器”，而是做那个在人与规则之间不断补足体验缺口的层：帮玩家更快进入角色，帮不会盘的人有台词可说，帮社恐玩家获得安全的表达支点，帮断节奏的房间重新热起来，帮搜证和盘问的碎片重新组织成可以推理的线。线上平台已经证明，玩家愿意为“稳定即时语音、快速匹配、系统说书人与公证、私聊和搜证体验”付费；而玩家研究又证明，**主持能力与角色承托**是体验成败的关键变量。把这两组事实放在一起，结论就是：LLM 不一定要取代人，但它非常适合补齐“主持能力不稳定”“玩家表达能力不均匀”“节奏断档”这些最真实的体验裂缝。citeturn3search0turn3search1turn30search2turn20search0

按玩家旅程拆开看，LLM 的价值点也很清楚：

| 游玩阶段 | 玩家真正想要什么 | 最适合的 LLM 角色 |
| --- | --- | --- |
| 入场与选角 | 快速理解风格、避免尴尬、不怕上手成本 | 选本助手、角色导读、破冰主持 |
| 阅读角色本 | 抓住自己能说什么、不能说什么、该隐藏什么 | 角色白话解释器、禁说提醒器 |
| 搜证 | 从大量碎片里更快找到方向 | 线索归类、时间线整理、矛盾点提示 |
| 盘问 | 既能自由问，又不至于问废 | NPC 扮演、追问建议、谎言检测提示 |
| 圆桌讨论 | 轮到我时知道该说什么，不卡壳 | 发言提纲、役职立场摘要、说话顺序建议 |
| 复盘与结算 | 得到 closure，而不是只看答案 | 个人视角复盘、关系线回放、情感后日谈 |

这一整张表背后有一个更重要的判断：**用户并不需要 LLM“全知全能”，用户需要的是它在关键时刻“托一下”**。这决定了产品设计应该优先做“低风险高感知”的介入点，而不是一上来追求“全局 AI 主持”。citeturn17search0turn20search0turn3search1turn30search2

## 单人模式怎么做

如果只看“玩法—能力耦合度”，单人模式是 LLM 最应该打深的战场。原因很简单：单人局的核心乐趣，本来就高度依赖**自由问答、角色扮演、节奏引导、个性化提示**，而这些正是 LLM 的强项。研究端已经有相当明确的技术积累：MMAgents 证明了在 Murder Mystery 这种复杂社会推理环境里，引入“adjacency pairs + next-speaker selection”的轮次控制机制，可以显著降低对话崩溃并提升信息共享与逻辑推理；ProTriPlay 则把交互戏剧拆成导演、编剧、演员三个 Agent，演员用 RAG 回忆相关交互，导演维护全局节奏和角色内在状态；Generative Agents 进一步给出“记忆—反思—计划”的经典架构，用来支撑长程角色一致性。citeturn9view1turn9view2turn32view0turn33view1turn22search2

产品端的信号也一致。Cluebound 把“对嫌疑人自由发问”做成日更侦探玩法；Arsenic 把“可自由盘问、角色会撒谎、躲闪、前后不一”直接写进卖点；Steam 上的《谜境·AI剧本杀互动剧场》则明确采用“AI NPC 驱动交互”的数字剧场思路。这说明用户已经开始接受这样一种单人推理体验：**核心不是读固定剧本，而是和角色动态交互，从相互试探里把真相挖出来**。citeturn29search7turn29search9turn30search1

真正有潜力的单人设计，并不是“让一个大模型同时扮演一切”，而是把能力拆层。一个更稳妥的设计是：**导演 Agent 管阶段和节奏，演员 Agent 管 NPC 语言与态度，编剧/场景 Agent 管环境反馈与叙事缝合，规则机维护真值与解锁条件**。ProTriPlay 的实证结果表明，这种三元结构比单 Agent 或双 Agent 更能维持叙事连贯、环境描写和场景过渡；而 ProTriPlay 还特别提出了一个很值得借鉴的点：角色除了“记住发生了什么”，还要维护一个显式的**内部状态**，例如当前情绪、与玩家关系、当前欲望和目标。也就是说，NPC 不应只有记忆库，还要有“**现在怎么想**”的状态机。citeturn32view0turn33view2

这会直接改变剧本杀单人模式的设计重点。对推理/还原本，LLM 最适合做的，是把盘问从“菜单选择题”升级为“开放审问”，再把回答落回到结构化线索板与时间线板上；对情感本，LLM 更适合承接“信件、梦境、独白、后日谈、关系修复”这些高情绪密度内容；对欢乐本，LLM 可以是抖包袱和接梗的人；对机制/阵营本，LLM 则应该收敛，只做 flavor text、提示和新手引导，因为平衡与结算才是核心。中国现有的本型分类和玩家动机研究恰好支持这种分法：越偏脑力/情绪/扮演，本就越适合用 LLM 放大；越偏阵营/机制/硬胜负，越应该让出控制权给显式规则。citeturn0search1turn0search2turn1search3turn17search0

文档里提到的“生成式 UI”也很有前景，但它的正确打开方式不是让模型直接写一堆不可控 HTML，而是让它生成**结构化组件意图**。Google 的 Generative UI 工作已经说明，现代模型在工具辅助下确实可以为任意 prompt 生成高质量自定义界面，而且在人类偏好评测里明显优于普通 markdown 文本；但同一篇工作也承认，速度仍然是显著问题，产品实现依赖工具访问、细致系统指令和后处理。因此，在剧本杀里，适合把生成式 UI 用在“线索板、时间线板、人物关系图、搜证清单、嫌疑矩阵”等有限组件集合上，而不是完全开放地生成整个前端。citeturn11view0turn10search0

## 多人模式怎么做

多数设计文档在多人模式里选择“让 LLM 退场”，这是对的，但只能对一半。对的地方在于：**LLM 不该当裁判**。国家层面的剧本娱乐管理已经明确要求剧本脚本备案、适龄范围、内容自审、未成年人保护和内容红线；线上平台的现实产品形态也说明，多人局的底层仍是房间管理、信息分发、公证、匹配和語音互动。因此，任何涉及回合合法性、线索可见性、投票、技能结算、身份权限的环节，都应该交给服务端权威状态机。citeturn18search1turn18search5turn30search2

但错的地方在于：**多人局不该让 LLM 完全退场**。因为多人局的真实痛点，往往不在规则判定，而在社交与节奏：有人不会演、有人不会盘、有人掉线、有人控场不足、冷场时没人接、陌生人局没有氛围、复盘时信息太碎、DM 水平不稳定。平台端已经把 DM、系统说书人与公证作为价值点卖了很多年；同时，中国市场也已经出现“剧狐 DM”这样的 DM 工具化产品，甚至在 2025 年版本更新里上线了“AI 开本助手”；另有《谜境·AI剧本杀互动剧场》这种更激进的 AI NPC 方案。这说明市场真实演化路径很可能不是“多人局完全无 AI”，而是“**多人局先出现大量 LLM Copilot，再局部出现 AI 替补与 AI 协演**”。citeturn30search2turn31search5turn31search1turn30search1

所以，多人模式里更合理的定位不是 “LLM as Judge”，而是 “LLM as Co-DM”。最值得做的几个角色是：第一，**气氛主持**，负责开场旁白、阶段转场、沉默催促、赛后收口；第二，**新手私教**，在不泄密的前提下用私聊给新玩家解释规则、提醒发言机会、帮他整理自己当前可说的观点；第三，**缺人替补**，只在必要时接某个 NPC、路人证人或临时位置，而不接管核心裁决；第四，**违规与内容风控助手**，先做提示与标注，再由规则机或人工决定是否处罚；第五，**复盘师**，按个人视角、阵营视角、时间线视角生成赛后材料。这个定位与官方平台对“DM、系统说书、公证”的现实做法是一致的，也与政策上“内容自审、适龄标注、内容红线”要求兼容。citeturn30search2turn18search1turn18search5

从本型看，多人模式里 LLM 的介入力度也应该有明显差异。阵营本和机制本的核心是平衡、技能结算和群体心理博弈，LLM 可以做热场和解释，但不该碰胜负；推理本和还原本更适合让 LLM 帮忙梳理公共信息、维护节奏和扮演边缘 NPC；情感本则特别适合在收尾阶段让 LLM 接手“个人后记、角色书信、隐藏视角补完”；恐怖本可以把 LLM 用在氛围文本和多模态旁白上，但必须加强内容边界和适龄控制。这个差异化设计，远比“多人模式统一不开 AI”更接近可落地产品。citeturn0search1turn0search2turn1search3turn18search5

## 运行时融合架构

如果要补上两份文档里真正的空白，核心不是再训练一个更会演的模型，而是把运行时工程改成“**状态机管真值，LLM 管表演**”。当前最有价值的外部启发非常一致：JSAI 2025 的工作用**状态迁移结构**来提升场景生成的一致性；STORY2GAME 用 LLM 生成动作的**前置条件与效果**，明确告诉游戏引擎“哪些状态该被跟踪和变化”；RPGBench 则把“结构化事件—状态表示”“变量更新检查”“事件机制验证”放进评测核心。它们共同指向一件事：只要内容会影响可玩性，LLM 输出就必须被翻译成**显式状态变化合同**。citeturn26view0turn14view0turn25view0

这意味着，NPC 的一次回答在系统里不应该只有一段文本，而应该至少拆成两条通道：一条是给玩家看的**戏内回答**；另一条是给引擎看的**结构化提案**。后者可以长成这样：

```json
{
  "speech": "昨晚九点我确实去过厨房，但只是去拿药。",
  "speech_act": "partial_admission",
  "claims": ["fact.kitchen.21:00", "fact.medicine"],
  "proposed_events": [
    {"type": "mark_contradiction", "target": "npc.maid.statement_03"},
    {"type": "candidate_unlock", "clue_id": "bottle_label"}
  ],
  "memory_writes": [
    {"scope": "public", "truth_status": "claim", "content": "npc says he went to kitchen at 21:00"}
  ],
  "confidence": 0.78
}
```

关键不在 JSON 长什么样，而在**谁说了算**。结构化输出可以借助 JSON Schema、工具调用与严格模式来保证字段可靠、类型安全、拒绝可检测；但真正的状态提交，必须由 DSL/规则层完成二次验证：当前是不是盘问阶段；这个 NPC 是否有权限说到该事实；该线索是否已经满足解锁条件；这段表述是否碰到适龄/内容红线；提议写入的 memory scope 是否越权。OpenAI 和 Anthropic 的官方文档都已经把“工具参数由 JSON Schema 定义、严格遵守 schema、可检测 refusal”做成了能力；但 OpenAI 也明确提醒，**字段正确不等于语义正确**。所以，最稳定的做法不是“让模型直接改 flags”，而是“让模型**提案**，让规则机**裁决**”。citeturn12search1turn12search2turn12search0

这也正好回答你提到的 visibility rule 问题。正确做法不是在 prompt 里写一句“不要剧透”，而是在数据层把所有事实分成至少五个作用域：**system truth、role private、group private、public confirmed、public rumor**。平台现有产品已经在实践类似的可见性差异：百变大侦探支持私聊与场景搜证；线上剧本杀平台本来就存在公共讨论、私聊、搜证展示这些不同通道。对 NPC 来说，记忆也必须按这个模型入库：玩家 A 在公开圆桌上说了谎，那么所有在场角色都可记；A 私聊骗了 B，那么只有 A、B 和可能的窃听角色能写入；系统判定出的隐藏真相，永远只存在于 system truth，除非解锁。这样，问题“玩家 A 刚对 B 撒的谎，C 问话时 B 该知道吗”就不再靠 prompt 瞎猜，而是变成事件作用域计算。citeturn3search1turn3search4turn33view1turn22search2

在记忆本身上，也不该只做“角色生平/动机/轨迹”的静态 RAG。ProTriPlay 和 Generative Agents 都提醒了一点：**记忆只回答“发生了什么”，内部状态才回答“我现在怎么想”**。对剧本杀 NPC 而言，这一点尤其重要。一个合格的嫌疑人不只是背设定，他还应该有当前压力值、对玩家的信任值、是否决定转移嫌疑、是否准备甩锅、是否开始防守过度等动态状态；这些状态不需要暴露给玩家，但必须参与回应生成。否则 NPC 会“记住过去”，却不会“在当下改变立场”。citeturn33view2turn33view1turn22search2

延迟预算也必须按交互类别来拆，而不是统一追求“大模型尽快答完”。HCI 和对话系统研究都给了很稳定的启发：0.1 秒、1 秒、10 秒是用户感知的经典阈值；进度反馈可以显著减少等待的负面感受；对聊天系统而言，等待解释还能提升透明度与信任；但不同人群、不同场景对速度的偏好并不一样，年轻用户更偏好快答，而偏陪伴/情绪类场景允许更有人味的节奏。对剧本杀来说，一个可执行的预算是：按钮确认和搜证面板回显应尽量压到 300ms 以内；NPC 盘问至少要在 1–1.5 秒内开始流式出首句；普通短答尽量在 3–5 秒内完结；复杂推理、阶段切换、总结和复盘则可以容忍 8–15 秒，但必须配合“思考中/检索中/整理证据中”的反馈。citeturn35view1turn35view0turn35view2turn35view3turn34view0

最后是失败兜底。真正可上线的方案，不能只做“提示词优化”，而要做**可检测、可回滚、可重放**。推荐的最小闭环是：所有模型提案先写入 event log 的 shadow 区；验证器检查 phase、visibility、unlock rule、内容安全；通过后再提交 authoritative state；若发现泄露未解锁信息，则优先采取“阻塞流式输出—重生成—替换成躲闪模板”的方式；若已出现在玩家端，则事件流必须支持撤销、补丁广播和赛后审计。之所以要这么重，不是因为模型一定会错，而是因为已经有研究直接证明：LLM 驱动的 NPC 会被 adversarial prompt injection 诱导“吐出本不该说的隐藏秘密”；而另一边，RPGBench 也说明仅靠 LLM 自身，很难长期保持可验证的机制一致性。citeturn7view0turn25view0turn12search1

## 产品路线与优先级

如果把“好不好玩”和“能不能上线”同时考虑，最合理的路线不是一步做到“AI 全托管剧本杀平台”，而是沿着**单人深融合 → 多人轻辅助 → 特定场景 AI 主持 → 创作运行闭环**这样的顺序推进。

第一阶段最值得做的是**单人推理/还原本**。原因是这类玩法对自然语言审问、嫌疑人记忆、线索收束、时间线整理的需求最强，公平结算的外部压力又相对最低；同时，国际上已经出现能够教育用户的自然语言审问产品，研究界也有成熟的 MMG、多 Agent 对话、RAG 记忆与内部状态方案可借鉴。这个阶段的目标不是追求 100% 开放世界，而是做出一个“自由审问 + 结构化线索板 + 可验证解锁”的完整闭环。citeturn29search7turn29search9turn9view1turn33view1turn25view0

第二阶段应该做**多人 Copilot DM**，而不是多人 AI Judge。也就是把 LLM 放在旁白、节奏、新手辅导、复盘、缺席替补、内容预警这些非裁决位上。这样做的好处是：一方面可以明显改善真实房间里最常见的体验问题，另一方面又不会立刻触碰规则公信力。市场上已经存在 DM 工具化尝试，政策层面对内容边界也很明确，这条路的组织风险和工程风险都明显低于“让 AI 直接判一整局”。citeturn31search5turn18search1turn18search5

第三阶段才是**在特定玩法里尝试 AI 主持**。优先顺序应当是：无主持派对型、单人/双人侦探型、边缘 NPC 驱动型，最后才是传统多人核心局。因为无主持和派对型产品天然就把“没有专业 DM 也能玩起来”当卖点，用户对 AI 主持的容忍度和期待值都更匹配；而传统核心多人局，用户对公平、公证、节奏、沉浸的心理阈值反而更高。citeturn29search2turn29search10turn29search11turn19search1turn29search3

第四阶段才值得回头做你文档里那种**创作期与运行期的闭环**：创作者在 scene DSL 里定义阶段、visibility、unlock、真相树、线索对象和可选 NPC 状态维度；系统在运行期收集玩家真实盘问日志、矛盾暴露节点、发言轮次分布、解锁路径、误导成功率；这些数据再反哺到微调和奖励建模里，专门训练“会守密、会周旋、会在适当时机泄露、但绝不越权”的 NPC。ACL 2026 那篇关于 MMG 的工作已经给出了很像的训练方向：先用角色化、多模态、多跳 reasoning 数据做 CoT finetune，再用 GRPO 做角色特定行为与抗欺骗强化。也就是说，你文档里的 “SFT + GRPO 抗欺骗” 不是空想，但它应该是**最后一层优化**，不是替代运行时规则约束的第一层。citeturn8view0

把整篇报告压成一句结论，就是：**剧本杀与 LLM 的最佳结合方式，不是把 LLM 变成无所不能的裁判，而是把它变成强表达、强陪玩、强承托、但永远受 DSL 和权威状态机约束的“戏剧层”**。单人局让它多演，多记，多引导；多人局让它多托，多补，多暖场；而决定可玩性的真值、权限、解锁与结算，始终应该留在可审计、可回放、可验证的规则系统里。外部产品、政策与研究三条线索，指向的都是同一个方向。citeturn25view0turn26view0turn12search1turn18search1turn32view0
